# Blueprint to Resource Map

This document answers one question: **how does Gentepede know which AWS services to provision, and which to skip, for each blueprint?**

If you're working on Spring Boot + Postgres, you don't need DynamoDB or Redis. If you're working on Ktor + DynamoDB, you don't need RDS. This document explains the mechanism that ensures the right services are created — and the wrong ones are not.

---

## The Short Answer

Every blueprint JSON file declares exactly which AWS resources it needs (the `awsResources` array). Gentepede reads this list, derives boolean flags, writes them to `terraform.tfvars`, and the Terraform template creates only the resources whose flag is `true`.

---

## The Full Chain (Step by Step)

### Step 1 — Blueprint declares its AWS resources

Each blueprint JSON in `src/main/resources/blueprints/` contains an `awsResources` array listing every AWS service the blueprint requires. For example, `springboot-postgres.json`:

```json
"awsResources": [
  { "type": "VPC",           "required": true },
  { "type": "ALB",           "required": true },
  { "type": "ECS_FARGATE",   "required": true },
  { "type": "RDS_POSTGRES",  "required": true },
  { "type": "KMS",           "required": true }
]
```

Compare with `ktor-dynamodb.json`:

```json
"awsResources": [
  { "type": "VPC",            "required": true },
  { "type": "ALB",            "required": true },
  { "type": "ECS_FARGATE",    "required": true },
  { "type": "DYNAMODB_TABLE", "required": true },
  { "type": "KMS",            "required": true }
]
```

The difference: `RDS_POSTGRES` vs `DYNAMODB_TABLE`. That single field difference is the source of truth.

### Step 2 — InfrastructureService derives toggle flags

`InfrastructureService.injectDataTierToggles()` in `InfrastructureService.kt` reads the blueprint's `awsResources` list and derives boolean flags:

```kotlin
val types = blueprint.awsResources.map { it.type }.toSet()

when (blueprint.templateFamily) {
    TemplateFamily.ecs -> {
        merged.putIfAbsent("enable_rds",      JsonPrimitive("RDS_POSTGRES"      in types))
        merged.putIfAbsent("enable_dynamodb", JsonPrimitive("DYNAMODB_TABLE"    in types))
        merged.putIfAbsent("enable_redis",    JsonPrimitive("ELASTICACHE_REDIS" in types))
    }
    TemplateFamily.eks -> {
        merged.putIfAbsent("enable_rds",      JsonPrimitive("RDS_POSTGRES"      in types))
    }
    TemplateFamily.lambda -> {
        // no toggles — lambda blueprints always provision all declared resources
    }
}
```

The logic is: if `RDS_POSTGRES` is in the blueprint's resource list → `enable_rds = true`. Otherwise `false`. No guessing, no naming conventions, no magic.

### Step 3 — Toggles are written to terraform.tfvars

The derived booleans land in `terraform.tfvars` in the project workspace (e.g. `~/.gentepede/workspaces/my-api/terraform.tfvars`):

```hcl
# Generated by Gentepede MCP — springboot-postgres blueprint

project_name     = "my-api"
container_image  = "123456789012.dkr.ecr.us-east-1.amazonaws.com/my-api:1.0.0"
certificate_arn  = "arn:aws:acm:us-east-1:..."

# Data-tier toggles — derived from blueprint awsResources
enable_rds      = true
enable_dynamodb = false
enable_redis    = false
```

### Step 4 — Terraform counts resources based on the flags

The shared `main.tf` uses `count = var.enable_X ? 1 : 0` on every data-tier resource block:

```hcl
resource "aws_db_instance" "postgres" {
  count = var.enable_rds ? 1 : 0   # ← 1 for springboot-postgres, 0 for everything else
  ...
}

resource "aws_dynamodb_table" "main" {
  count = var.enable_dynamodb ? 1 : 0   # ← 1 for ktor-dynamodb only
  ...
}

resource "aws_elasticache_replication_group" "redis" {
  count = var.enable_redis ? 1 : 0   # ← 1 for fastapi-redis only
  ...
}
```

When `count = 0`, Terraform creates no resources from that block and emits no costs.

## Why Each Stack Needs Its Specific Data Tier

### Spring Boot + RDS PostgreSQL

Spring Boot's primary data access layer is Spring Data JPA backed by Hibernate. Hibernate is an ORM (Object-Relational Mapper) — it translates Java objects to SQL rows and vice versa. This ORM model requires a relational database with ACID transactions, foreign keys, and SQL query support. PostgreSQL is the canonical choice.

DynamoDB's NoSQL model is incompatible with Hibernate's relational assumptions. Using DynamoDB from Spring Boot requires a completely different SDK (DynamoDB Enhanced Client or Spring Data DynamoDB) and a different data modeling approach. The springboot-postgres blueprint is for teams using the standard Spring Data JPA stack.

### Ktor + DynamoDB

Ktor is a lightweight async Kotlin HTTP framework. It pairs naturally with the AWS SDK v2's async DynamoDB client, which uses Kotlin coroutines. DynamoDB requires no connection pool (HTTP-based), no schema migrations, and scales automatically. This makes it a low-friction choice for Kotlin microservices that don't need relational queries.

If the Ktor service needs relational data, fork the blueprint and substitute RDS (see [docs/10-adding-blueprints.md](10-adding-blueprints.md)).

### FastAPI + ElastiCache Redis

FastAPI is a Python async web framework often used for ML inference endpoints, recommendation systems, or high-throughput APIs where response caching is critical. Redis provides sub-millisecond in-memory reads, making it ideal for:
- Caching expensive computation results (e.g. ML model inference)
- Session storage across multiple FastAPI instances
- Rate limiting counters
- Pub/sub between services

Redis is not a replacement for a relational database — it is a cache layer in front of it. If FastAPI also needs PostgreSQL, the blueprint can be extended.

### Node.js + Lambda + S3 + CloudFront

Node.js serverless APIs (Lambda) commonly need two things: compute (Lambda) and persistent storage (S3). Lambda itself is stateless — each invocation is isolated. S3 provides durable object storage for user uploads, generated files, and application data.

CloudFront in front of Lambda serves two purposes:
1. Global edge presence: API Gateway is regional; CloudFront puts the API closer to users worldwide
2. Custom domain + TLS: CloudFront supports custom domain names with ACM certificates

No relational database is included because Lambda cold start constraints and connection pool exhaustion make RDS a poor fit for Lambda. For Lambda + database, use DynamoDB (connection-free) or Aurora Serverless.

### Node.js + EKS + S3 + CloudFront

Same rationale as Lambda for S3 + CloudFront, but the compute layer is EKS instead of Lambda. Use this blueprint when:
- The Node.js service needs always-on container processes (WebSockets, background workers)
- You want Kubernetes tooling (HPA, NetworkPolicy, Helm)
- The service is part of a larger microservice architecture already on EKS

## How to Add a New Data Tier

If a new blueprint needs a data tier that doesn't exist yet (e.g. Aurora Serverless, DocumentDB):

1. Add a new resource type string to the blueprint's `awsResources` array (e.g. `"AURORA_SERVERLESS"`)
2. Add a new variable `enable_aurora` to the relevant `variables.tf`
3. Add the new resource blocks to `main.tf` gated with `count = var.enable_aurora ? 1 : 0`
4. Add the toggle derivation to `injectDataTierToggles()` in `InfrastructureService.kt`:
   ```kotlin
   merged.putIfAbsent("enable_aurora", JsonPrimitive("AURORA_SERVERLESS" in types))
   ```
5. Run the blueprint verifier: `./gradlew shadowJar && java -cp build/libs/gentepede-mcp-all.jar com.gentepede.ci.BlueprintVerifierKt --blueprint <new-id> --project test`

See [docs/10-adding-blueprints.md](10-adding-blueprints.md) for the complete step-by-step guide.

## AWS Service Glossary

For readers unfamiliar with these services — a plain-English summary of each one used across the blueprints.

| Service | What it is | Why it's in every blueprint |
|---|---|---|
| **VPC** | Virtual Private Cloud — an isolated network in AWS. Your resources get private IP addresses and are invisible to other AWS customers. | All compute and data resources live inside it. Nothing is exposed to the internet except through explicitly configured entry points (ALB, CloudFront). |
| **KMS** | Key Management Service — manages cryptographic keys. You create a key; AWS stores it in hardware security modules. | Every blueprint uses a dedicated KMS key to encrypt data at rest (RDS, DynamoDB, S3, CloudWatch Logs, Lambda env vars). Without KMS, data is stored in plaintext. |
| **NAT Gateway** | Network Address Translation Gateway — gives private subnet resources outbound internet access without giving them a public IP. | ECS tasks and EKS nodes live in private subnets but need internet access to pull Docker images from ECR. NAT Gateway provides that path while keeping them unreachable from the internet. |
| **ALB** | Application Load Balancer — Layer-7 HTTP/HTTPS reverse proxy. Terminates TLS, distributes requests across healthy targets. | ECS and EKS blueprints use an ALB as the single internet-facing entry point. Your ACM certificate lives here. All other resources have no public IP. |
| **ECS Fargate** | Elastic Container Service with Fargate launch type — serverless containers. You provide a Docker image; AWS manages the host fleet. | The compute layer for spring-boot, ktor, and fastapi blueprints. No EC2 instances to manage. |
| **EKS** | Elastic Kubernetes Service — managed Kubernetes control plane. AWS runs etcd, API server, scheduler. You provide worker nodes. | The compute layer for eks blueprints. Enables full Kubernetes tooling: Helm, HPA, NetworkPolicy, operators. |
| **RDS PostgreSQL** | Relational Database Service running PostgreSQL — managed SQL database with automatic backups, patching, and Multi-AZ failover. | The relational data tier for Spring Boot + JPA blueprints. Hibernate/Spring Data JPA connects to it as a standard PostgreSQL endpoint. |
| **DynamoDB** | Managed NoSQL key-value database with millisecond latency at any scale, no servers to manage. | The data tier for Ktor blueprints. No connection pools, schema-free, scales to any throughput. |
| **ElastiCache Redis** | Managed in-memory Redis — stores data entirely in RAM for sub-millisecond reads. | The caching and session storage tier for FastAPI blueprints. 10–100× faster than database reads for hot data. |
| **Lambda** | Serverless function compute — runs code in response to events, billed per invocation. No servers. | The compute layer for Node.js serverless blueprints. Costs near zero at low traffic; scales to millions of requests automatically. |
| **API Gateway** | Managed HTTPS endpoint that invokes Lambda functions. | Exposes the Lambda function to the internet as a standard HTTPS API. Without it, Lambda can only be invoked via AWS SDK/CLI. |
| **S3** | Simple Storage Service — durable object storage at any scale. | Object storage for Lambda and Node.js EKS blueprints. Stores uploads, generated files, and static assets with 11 nines of durability. |
| **CloudFront** | Global CDN with 400+ edge locations. Terminates TLS at the edge. | Sits in front of API Gateway (Lambda) and S3 (EKS) to reduce latency globally and enable custom domain names. |

For Helm and Kubernetes concepts (Deployment, Service, HPA, NetworkPolicy, ResourceQuota), see [docs/06-kubernetes-guide.md](06-kubernetes-guide.md).
