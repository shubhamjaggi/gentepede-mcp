# Dual-Mode Deployment Guide

## The Two Modes

Controlled by the `GENTEPEDE_MODE` environment variable:

| Variable value | AWS target | Kubernetes target | State storage |
|---|---|---|---|
| `LOCAL` (default) | LocalStack at http://localhost:4566 | kind cluster `gentepede-local` | Local `terraform.tfstate` file |
| `PRODUCTION` | Real AWS (uses env credentials or profile) | Real EKS cluster | S3 + DynamoDB remote backend |

The mode is read once at server startup. Change it by updating `GENTEPEDE_MODE` in `claude_desktop_config.json` and restarting Claude Desktop.

---

## LOCAL Mode

### What is LocalStack?

LocalStack is a Docker image that emulates most AWS services locally. It implements the same HTTP API as real AWS (`http://localhost:4566`). Terraform sends the same API calls as it would to real AWS — LocalStack intercepts and responds with simulated resource state.

Cost: **free** (no AWS account charges for local development).

### Starting LocalStack

```bash
docker run -d --name localstack \
  -p 4566:4566 -p 4510-4559:4510-4559 \
  localstack/localstack

# Verify it's healthy
curl http://localhost:4566/_localstack/health
```

Expected response: `{"services": {"s3": "running", "ec2": "running", ...}}`

### LOCAL providers.tf (generated by InfrastructureService)

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "= 5.82.0"    # from blueprint.terraformProviderVersion
    }
  }
  # No backend block — state stored in local terraform.tfstate
}

provider "aws" {
  region                      = var.aws_region
  access_key                  = "test"     # LocalStack accepts any key
  secret_key                  = "test"
  skip_credentials_validation = true       # No real AWS credentials to validate
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3          = "http://localhost:4566"
    ec2         = "http://localhost:4566"
    iam         = "http://localhost:4566"
    rds         = "http://localhost:4566"
    dynamodb    = "http://localhost:4566"
    ecs         = "http://localhost:4566"
    # ... all services point to LocalStack
  }
}
```

### kind for EKS Blueprints (LOCAL mode)

**What is kind?** Kubernetes IN Docker — a tool that creates Kubernetes clusters using Docker containers as nodes. It is free, requires only Docker Desktop, and creates clusters in seconds.

**Why doesn't Gentepede auto-create the kind cluster?** InfrastructureService checks for the cluster with `kind get clusters` and aborts with instructions if `gentepede-local` is not found. This is intentional:
1. A kind cluster is not automatically torn down between Gentepede invocations — it persists across sessions.
2. Auto-creating it would hide the cluster lifecycle from the user, making it harder to manage.
3. It separates the one-time cluster setup from the repeatable generate → validate → apply workflow.

**Creating the cluster:**
```bash
# Using the config generated by generate_infrastructure_package:
kind create cluster --name gentepede-local \
  --config ~/.gentepede/workspaces/{project_name}/kind-config.yaml

# Verify:
kind get clusters        # → gentepede-local
kubectl get nodes        # → shows 1 control-plane + 2 worker nodes
```

After creation, `kubectl` automatically has a context named `kind-gentepede-local`. Helm targets this context for LOCAL deployments.

## PRODUCTION Mode

### PRODUCTION providers.tf (generated by InfrastructureService)

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "= 5.82.0"
    }
  }

  backend "s3" {
    bucket         = "my-project-tfstate"       # must exist before first apply
    key            = "gentepede/my-project/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "my-project-tfstate-lock"  # must exist before first apply
    encrypt        = true
  }
}

provider "aws" {
  region  = var.aws_region
  profile = var.aws_profile    # uses your AWS CLI named profile
}
```

### Annotation: LOCAL vs PRODUCTION providers.tf differences

| Feature | LOCAL | PRODUCTION |
|---|---|---|
| `backend` block | Absent (local state) | Present (S3 + DynamoDB) |
| `access_key` | `"test"` (LocalStack) | Not set (uses env vars or profile) |
| `skip_credentials_validation` | `true` | Not set (validates real credentials) |
| `endpoints` | All pointing to `localhost:4566` | Not set (uses real AWS endpoints) |
| `profile` | Not set | `var.aws_profile` |

### What PRODUCTION Requires

**1. AWS credentials** — set via environment variables or named profile:
```bash
# Option A: environment variables
export AWS_ACCESS_KEY_ID=AKIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_DEFAULT_REGION=us-east-1

# Option B: named profile in ~/.aws/credentials
export AWS_PROFILE=my-profile
```

**2. IAM permissions** — the IAM principal must have permissions to create every resource type in the blueprint. Minimum set for `springboot-postgres`:
```
ec2:* (VPC, subnets, security groups, NAT gateway, EIP, route tables)
elasticloadbalancing:* (ALB, listeners, target groups)
ecs:* (cluster, task definitions, services)
rds:* (DB instances, subnet groups)
iam:CreateRole, iam:AttachRolePolicy, iam:PutRolePolicy, iam:PassRole
kms:CreateKey, kms:CreateAlias, kms:DescribeKey
logs:CreateLogGroup, logs:PutRetentionPolicy
s3:CreateBucket, s3:PutBucketEncryption, s3:PutBucketVersioning, s3:PutPublicAccessBlock
```

**3. Remote state prerequisites** — before the first PRODUCTION `terraform apply`:
```bash
# Create S3 state bucket
aws s3api create-bucket \
  --bucket {project_name}-tfstate \
  --region us-east-1

# Enable bucket encryption
aws s3api put-bucket-encryption \
  --bucket {project_name}-tfstate \
  --server-side-encryption-configuration '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}'

# Create DynamoDB lock table
aws dynamodb create-table \
  --table-name {project_name}-tfstate-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

### Credential Pre-flight (PRODUCTION only)

Before every PRODUCTION operation that contacts AWS (plan, apply, drift, destroy), Gentepede runs:
```bash
aws sts get-caller-identity
```

The response is parsed and prepended to the tool output:
```
Acting as: arn:aws:iam::123456789012:role/deploy-role
Account:   123456789012
User ID:   AROAXXXXXXXXXXXXXXXXX
```

If this call fails, the operation aborts immediately with a clear error. You never reach Terraform without confirmed identity. This prevents silent "wrong credentials" failures where Terraform runs with no permissions and produces confusing errors.

Note: `validate_infrastructure_package` does NOT run this pre-flight. It is pure static analysis (terraform validate + checkov + kube-score) that makes zero AWS API calls.
